## Problem Statement

Millions of non-engineers are shipping real products using AI coding tools (Claude Code, Cursor, Lovable, Bolt). They can build fast, but they ship blind: no analytics, no monitoring, no understanding of what users do or whether the system is healthy.

Even when they add tracking (via PostHog wizard, Claude Code, etc.), the metrics have no descriptions, no business context, no explanations. The tracking is shallow, generic, and not personalized to the product's actual business model, funnel, or user journey.

The bottleneck isn't installing an SDK. PostHog's wizard does that in 90 seconds. The bottleneck is the THINKING work: deciding what to track, why it matters, what normal looks like, and what to do when something breaks. That thinking work takes even an experienced PM about a week.

## Demand Evidence

- **Competitive intel:** Conversations with Resolve.ai and Traversal users confirmed that AI SRE/analytics products fail primarily due to lack of context. The AI can query data, but it doesn't know what the data means.
- **Own pain:** Founder (PM with multiple analytics setups at prior companies) needs tracking for Foggy right now but hasn't done it because "it requires time to define what best practice is." If an experienced PM procrastinates, a vibe-coder with zero analytics background will never do it unprompted.
- **Validated blind spot:** Vibe BI published "The Vibe Coder's Blind Spot: Data After Launch" confirming most vibe-coded products go live without structured logging, event tracking, or dashboards. Analytics is always "add later."
- **No direct vibe-coder conversations yet.** Demand is thesis-driven + own pain + competitive intel, not validated with target customers.

## Status Quo

- **Flying blind:** Most vibe-coders have zero analytics or monitoring
- **Platform defaults:** Some check Vercel analytics, Stripe dashboard, Supabase metrics — whatever their stack gives for free
- **AI-assisted but shallow:** Asking Claude Code "add analytics" installs PostHog SDK + a few generic capture() calls, but doesn't generate a tracking plan, doesn't explain what to track for THIS business, and doesn't produce metric descriptions
- **Datafast and alternatives:** Pre-built dashboards (Datafast, Sleek, GoatCounter, Umami, Plausible) exist but are one-size-fits-all, not personalized, not agent-consumable
- PostHog has wizard (installs SDK), AI plugin (27 tools for querying/managing), skills repo (under construction). All developer-first. None generate a personalized tracking plan or semantic layer.

## Target User & Narrowest Wedge

**Target user:** Somewhat technical founder/builder who ships with AI coding tools. They know analytics exists, understand concepts like funnels and retention, but haven't done the thinking work of deciding what to track for THEIR specific product. They want the planning work done for them, not just the coding work.

**Not the target (yet):** Completely non-technical person who's never heard of PostHog. And not the enterprise SRE buyer.

**Persona:** Can follow a PostHog tutorial, has deployed an app to Vercel or similar, uses AI coding tools daily. Knows what "conversion rate" and "error rate" mean but has never written a tracking plan or configured alerts. Can implement the plan if it tells them exactly what to do.

**Narrowest wedge:** A single Claude Code skill that scans the codebase, asks 5-7 business questions, and generates `tracking-plan.md` — a personalized tracking plan with metric descriptions, thresholds, and actions. No tracking code. No dashboard. Just the plan.

The plan is the product. Everything else is implementation that follows from it.

## Constraints

- Solo founder (PM, not engineer) building with AI coding tools
- No infrastructure budget for v1 — the skill is a standalone artifact, no server, no database
- Must be open source (the skill is the wedge, not the product)
- Must work in Claude Code AND Cursor (the two primary AI coding tools the founder uses)
- PostHog is moving fast in this space — speed matters more than completeness

## Premises

1. **The semantic layer is the bottleneck.** AI agents fail at analytics/observability not because they can't query data, but because they don't know WHAT to query or what the data means in business context.
2. **The plan is more valuable than the code.** A personalized tracking-plan.md is the core product. Tracking code is a commodity that PostHog/Claude Code can generate from the plan.
3. **The target user knows they need analytics but won't set it up.** Somewhat technical, uses AI tools, hasn't done the thinking work.
4. **An AI coding skill is the right distribution channel.** Users discover and run this inside Claude Code, Cursor, or similar.
5. **PostHog might build suggested events for their own platform, but won't build a cross-tool semantic layer.** The moat is being vendor-agnostic — spanning PostHog + OTel/Grafana + Sentry in one plan. (Revised after cross-model challenge.)

## Cross-Model Perspective

An independent Claude subagent reviewed the session context cold. Key findings:

- **Steelman:** "If the semantic layer becomes the standard way AI agents understand what a product does, Foggy owns the most defensible layer in the AI-native observability stack — the meaning layer between raw data and automated decisions."
- **Key insight:** The founder's own procrastination IS the product validation. An experienced PM who hasn't set up analytics for their own product proves the thinking work is the hard part, not the tooling.
- **Challenged premise #5:** PostHog is one sprint away from "suggest what to track." The only moat is if the semantic layer is cross-tool (PostHog + Grafana + Sentry), not PostHog-specific. Premise was revised in response.
- **Prototype suggestion:** Single skill, 5 questions, one markdown file. Test on Foggy first. "Does the generated plan make someone say 'yes, that's exactly what I should be tracking, and I wouldn't have written this myself'?"

## Approaches Considered

### Approach A: The Minimum Skill (Minimal Viable) -- CHOSEN
Single Claude Code skill. Scans codebase, asks 5 business questions, outputs tracking-plan.md. Open source. No tracking code, no OTel, no dashboard. Test on Foggy first, then share with beta users from vibe-coding communities.
- Effort: S (days of AI-assisted development)
- Risk: Low
- Pros: Ships fast, validates core thesis, zero infrastructure
- Cons: No visible output for users (just a markdown file), no tracking code means plan sits unused
- Completeness: 4/10

### Approach B: Cross-Tool Semantic Layer (Original concept, trimmed)
Two skills: PostHog analytics + OTel/Grafana. Each generates semantic layer AND tracking code committed to repo. tracking-plan.md spans both. Open source. No dashboard.
- Effort: M (3-4 weeks)
- Risk: Medium
- Pros: Cross-tool is the PostHog differentiator, users get working tracking code not just a plan
- Cons: 2x scope, OTel skill requires infra knowledge

### Approach C: Semantic Layer + Simple Dashboard (Alternative concept, phased)
Same as A but: plan format is machine-readable from day 1, AND generates a single-page dashboard that visualizes key metrics. Not a component library — one generated page.
- Effort: M (3-4 weeks)
- Risk: Medium-High
- Pros: Users get something they can SEE, dashboard is the "whoa" moment
- Cons: Dashboard quality risk, requires PostHog API integration, more scope

## Recommended Approach

**Approach A.** The acid test is whether the plan is good enough that someone says "yes, that's exactly what I should be tracking." If it is, B and C follow naturally. If it isn't, no dashboard saves you.

**Phasing to cross-tool moat:** Approach A validates plan quality using PostHog analytics only. The revised premise (#5) identifies the cross-tool semantic layer as the real moat. Approach B (adding OTel/Grafana) delivers this moat. The sequence is: A validates the thinking → B delivers the differentiation → C delivers the visible artifact. Each step requires the previous one to succeed.

**Contingency if plan quality is low:** If the generated plan for Foggy is mediocre compared to what the founder would write by hand, iterate on: (a) the question set (are 5 questions enough?), (b) the system prompt (does the AI understand what a good tracking plan looks like?), (c) codebase scanning depth (is it reading the right files?). If after 3 iterations the plan is still not useful, the premise "AI can generate the thinking work" is falsified, and the product direction needs to change.

**Codebase scanning strategy:** The skill reads files via Claude Code's context window. Priority order: (1) routes/pages (app structure), (2) auth/user model (user lifecycle), (3) package.json/pyproject.toml (stack detection, existing analytics), (4) payment integration files, (5) existing analytics calls. Heuristic: if tree output exceeds ~200 lines, switch to targeted reads. Total file content should stay under ~50K tokens to leave room for the system prompt, questions, and plan generation. Read file headers/exports first, pull full implementations only for analytically relevant files. For monorepos: ask the user which service to focus on.

The founder's plan is: ship A, test internally on Foggy, learn, and expand to B and C if the skill produces genuinely useful plans.

**Plan quality definition:** A plan is "good enough" when (a) each metric has a name, description, expected range, and actionable red flag, (b) metrics map to PostHog event schemas that can be directly implemented, (c) a PM reading it says "this covers what I would track," and (d) an AI agent reading it can answer "what happened to activation this week?" by querying the right PostHog events.

## Business Questions (Draft, 5 core)

These are asked interactively in the Claude Code conversation:

1. **What does your product do?** (Detects product type: SaaS, marketplace, API, mobile, etc.)
2. **How do users pay you?** (Subscription, freemium, one-time, ad-supported, free/pre-revenue)
3. **What does a successful user look like?** (The activation moment: "they create their first project," "they connect their Stripe account," "they invite a teammate")
4. **Where do your users come from?** (Organic search, social, referral, ads, Product Hunt, etc.)
5. **What's your biggest worry right now?** ("Users sign up but don't come back," "I don't know if the app is slow," "I can't tell which features people use")

The skill also auto-detects from codebase: tech stack, routes/pages, auth system, payment integration, existing analytics, deployment platform.

## Success Criteria

**Validation metrics (does the plan work?):**
1. **Internal acid test:** Run the skill on Foggy. Compare AI-generated plan to the hand-written plan (from The Assignment). Does it cover the same metrics? Is it as specific? Would you actually implement it?
2. **External acid test:** Share generated plans with 5+ builders who shipped products. Do they say "yes, this is what I should be tracking" without modification?

**Distribution metrics (does anyone care?):**
3. **Adoption:** 10+ people actually run the skill and generate a plan within 4 weeks of launch.
4. **Engagement:** At least 3 people implement tracking code based on the generated plan (not just generate it and forget it).
